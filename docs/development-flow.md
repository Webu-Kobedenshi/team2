# 開発の具体的な流れ

## 基本の流れ

1. Issue を確認する
2. ブランチを作る
3. 実装する
4. 手元で確認する
5. Pull Request を作る
6. レビューを受ける
7. マージする

### Issue を確認する

作業を始める前に、担当する Issue の内容を確認します。

- 何を作るか
- どこまでできたら完了か
- 画面や操作に関係する変更か
- 分からないことが残っていないか

分からないことがある場合は、実装を始める前にコメントや相談で確認してください。Issue の内容や作業範囲で迷った場合も、自分だけで判断せず相談してください。
もちろん実装途中で分からないところが出てきた場合も、遠慮なく聞いてください。

### ブランチを作る

作業ごとにブランチを分けます。

```bash
git switch -c feature/<issue-number>-short-title
```

例: issue numberが1で開始ボタンを実装するタスク

```bash
git switch -c feature/1-start-button
```

### 実装する

担当する Issue の完了条件に必要な変更だけ行います。

- Issue と関係ない修正は同じ PR に混ぜない
- 途中で別の問題を見つけたら、コメントや相談で共有する
- 何を直せば完了か迷ったら、実装を広げる前に確認する

### 手元で確認する

PR を作る前に、変更したところを自分で確認します。

- エラーが出ていないか
- formatter を通したか
- linter を通したか
- 仕様通りに動くか
- 他の機能が壊れていないか

など。
コマンドは `mise.toml` にあります。

```bash
mise run format:frontend
mise run check
```

### Pull Request を作る

PR には、何を変更したか、なぜ変更したか、などを書きます。

PR 作成画面にはテンプレートが出るので、空欄を埋めてください。分からないところは「ここが不安です」「ここを見てほしいです」と書いて構いません。

### レビューを受ける

レビュー依頼を投げて、他のメンバーにレビューしてもらいます。
コメントされていたら内容を確認し、修正、返信します。
返信したら「返信しました。」と一言連絡しましょう。

レビューの詳しい見方やコメントの書き方は [review.md](review.md) を見てください。

### マージする

PR が approve されたらマージしましょう。
マージ出来たらそのタスクは完了です。
お疲れさまでした。
